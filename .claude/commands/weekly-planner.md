# Weekly Planner Skill

Generates your structured weekly markdown note for the upcoming week.

This is a Tier 2 skill -- it requires an agentic tool (like Claude Code) with access to your actual files and integrations. In the browser, use `weekly-update` instead and fill in the template manually.

Before running this skill for the first time, configure your sources in `CLAUDE.md`:
- Where your weekly notes live (folder path)
- Where your projects live (folder path)
- Your documentation platform collection or URL (e.g., a Notion database ID)
- Your meeting notes source (e.g., Granola collection ID)
- Where your decisions log lives

---

## Step 1 -- Pull sources in parallel

Fetch all three sources at the same time:

### A. Documentation platform (last 3 weekly reports)
Pull your last 3 weekly ops reports from your documentation platform (configured in CLAUDE.md).

From your section only, extract:
- Goals from last week (with completion status)
- Goals for next week (starred items = top priorities)
- Experiments section
- Any bad / worry items that need follow-up

### B. Current week's markdown note
1. List files in your `Weekly/` folder to identify the current week file (highest date).
2. Read the file.
3. Extract:
   - All **incomplete** tasks `[ ]` -- these become carryovers
   - All **completed** tasks `[x]` -- for context only, do not carry forward
   - Meeting Actions with incomplete items
   - Any notes or key decisions

### C. Meeting notes (this week)
Pull meeting notes from your configured meeting notes source (e.g., Granola).

Filter to this week's meetings only.

Extract all next steps and action items that belong to you.

---

## Step 1b -- Build the tag inventory

Before composing the note, scan recent weekly notes to collect recently used tags.

1. List the 4 most recent weekly notes.
2. Read each.
3. Extract inline tags from the body: any token matching `#word` or `#word/subword`.
4. Deduplicate and store as `existing_tags[]`.

This list is used in Step 3 when assigning tags to Weekly Goals.

---

## Step 2 -- Determine next Monday's date

Calculate the date of next Monday (the week starting after the current Friday).
This becomes the filename: `YYYY-MM-DD.md`

---

## Step 3 -- Compose the weekly note

Use this structure:

```markdown
# Week of [Mon DD] - [Fri DD], YYYY

## Weekly Goals
- [ ] ⭐ [top priority 1 from your docs "goals for next week"] #tag
- [ ] ⭐ [top priority 2 if starred] #tag
- [ ] [other goal] #tag

## Carryover from Last Week
- [ ] [incomplete tasks from current week's note]
- [ ] [any in-progress items from docs goals last week]

## Meeting Actions

### [Meeting Name]
- [ ] [action item from meeting notes]

## Follow-ups
- [ ] [ongoing experiments or items to monitor]

## Experiments to Monitor
- [Experiment name] ([status])

## Daily Log

### Monday YYYY-MM-DD
- [ ] 

### Tuesday YYYY-MM-DD
- [ ] 

### Wednesday YYYY-MM-DD
- [ ] 

### Thursday YYYY-MM-DD
- [ ] 

### Friday YYYY-MM-DD
- [ ] 

## Notes

### Key Decisions from This Week
- **[Decision name]:** [brief summary]

### Wins from This Week
- [win 1]

### Carry-Forward Watch List
- **[item]** -- [brief context]

---
*Generated by Operator OS on [today's date]*
```

### Composition rules

- **Weekly Goals**: Take directly from your docs "Goals for next week". Preserve priority markers (⭐). For each goal, assign one inline tag using existing tags where a match exists, or create a new lowercase hyphenated tag.
- **Carryover**: Only include tasks that were explicitly unchecked `[ ]` in the current week's note OR marked in-progress in your docs. Do NOT carry forward completed items.
- **Meeting Actions**: Group by meeting. Only include actions that are yours (not other team members').
- **Follow-ups**: Ongoing experiments, monitoring tasks, pending decisions.
- **Experiments to Monitor**: Pull from your docs experiments section and any new ones from meetings.
- **Key Decisions**: From your docs "bad/worries" sections and meeting summaries -- things that were resolved this week.
- **Wins**: Concrete outcomes with data where available.
- **Carry-Forward Watch List**: Unresolved issues from worries/bad sections that need ongoing attention.

---

## Step 4 -- Save to file

Save the note to: `Weekly/YYYY-MM-DD.md` (next Monday's date).

---

## Step 4b -- Extract decisions and append to decisions log

### Find decisions from this week's sources

Scan fetched content for decisions. A decision is any explicit choice, direction, or commitment made -- not a task or observation.

Look for phrases like: "we decided", "agreed to", "going with", "will move forward with", "confirmed", "ruling out", "not doing."

For each decision, capture:
- **Title**: short descriptive name (5 words max)
- **Date**: when the decision was made
- **Context**: 1-2 sentences on why the decision was needed
- **Decision**: 1-2 sentences on what was decided
- **Source**: meeting name or doc date

### Deduplicate against existing log

Read `Context/decisions-log.md`. Skip any decision with an exact title match. Append everything else.

### Append format

```markdown
---

### [Decision Title]
- **Date:** YYYY-MM-DD
- **Source:** [Meeting name or doc reference]
- **Context:** [Why this decision was needed]
- **Decision:** [What was decided]
```

---

## Step 4c -- Resolve cross-references and create project stubs

After saving the weekly note, extract all `[[cross-references]]` from it and ensure each has a corresponding file in `Projects/`.

For each referenced project name with no existing file, create `Projects/[Name].md` with this template:

```markdown
# [Initiative Name]

**Status:** Active
**First referenced:** YYYY-MM-DD
**Weekly note:** YYYY-MM-DD

## Goal


## Open Questions


## Key Decisions


## Next Action
```

---

## Step 4d -- Flag stale watch list items

Compare the Carry-Forward Watch List with the previous two weeks' notes. Any item appearing in all three weeks gets a `[!]` marker:
- Example: `- **[!] Q2 pricing decision** -- awaiting leadership alignment; investigation ongoing`

---

## Step 5 -- Confirm to the operator

Reply with a brief summary:
- Filename saved
- Number of goals pulled
- Number of carryover tasks
- Number of meeting actions
- Tags assigned per goal (which were reused vs. newly created)
- Number of decisions appended (list their titles)
- New project stubs created (list names) -- or "none"
- Stale watch list items flagged with `[!]`
- Any items needing attention (kill decisions, blockers)

One short paragraph plus bullet list. Do not re-paste the full note.

After delivering the summary, ask: "Anything to add to the error log this week?" If the operator provides items, append them to `Context/error-log.md` using the format at the top of that file. If nothing to add, close out.

---

## Terminology rules

Apply terminology rules from the operator's `CLAUDE.md`. Defaults:
- No em dashes (-- or ---) or double dashes
- Do not add commentary or coaching to the note -- just structure and tasks
- Keep the tone of the note factual and direct
